## 任务 A：关键架构决策（6 个）

### 决策 1：知识表示范式（Graph 是否为"主引擎"还是"辅助解释层"）

**推荐方案**：**Graph-first（图谱主导推理）+ RAG 作为证据补充**

| 方案 | 说明 |
|------|------|
| A: Graph-first（你当前方案） | 图谱主导推理 |
| B: RAG-first | 向量检索为主，图谱仅做解释 |
| C: Hybrid but LLM-driven | 路径由 LLM 生成，图谱仅校验 |

**权衡**：
| 维度 | Graph-first | RAG-first |
|------|:----------:|:---------:|
| 可解释性 | ✅ 强（路径显式） | ❌ 弱 |
| 可控性 | ✅ 高 | ❌ 低 |
| 构建成本 | ❌ 高 | ✅ 低 |
| 泛化能力 | ❌ 受限 | ✅ 强 |

**为什么推荐 Graph-first**：
这是一个作品集项目，不是 recall benchmark 项目。你真正要展示的是"我可以构建可计算的医学推理结构"，而不是"我能调一个更好的 embedding + reranker pipeline"。如果你用 RAG-first，本质上你在做"医疗版 ChatGPT wrapper"；而 Graph-first 是"医学 reasoning engine"。这是面试价值的分水岭。

---

### 决策 2：图数据库选型（SQLite vs Neo4j vs 内存图）

**推荐方案**：**继续使用 SQLite + 递归 CTE（当前方案）**

| 维度 | SQLite | Neo4j | NetworkX |
|------|:-----:|:-----:|:--------:|
| 部署复杂度 | ✅ 极低 | ❌ 高 | ✅ 低 |
| 可解释性 | ✅ SQL 可审计 | ✅ 强 | ❌ 不持久 |
| 扩展性 | ⚠️ 中等 | ✅ 强 | ❌ 弱 |
| MVP 速度 | ✅ 快 | ❌ 慢 | ✅ 快 |

**为什么 SQLite 是正确选择**：
你这个系统的真实规模是：节点 30-100，边 50-200，深度 ≤3。这不是"图数据库问题"，而是"带递归查询的结构化知识表"。引入 Neo4j 在这个阶段属于典型过度工程 + 减少可控性（增加黑箱）。

---

### 决策 3：路径搜索策略（盲搜 vs 约束推理）

**推荐方案**：**"层级约束 + 关系白名单"的有向路径搜索**

| 方案 | 说明 |
|------|------|
| A: 盲 DFS/BFS（当前隐含方案） | 无约束搜索 |
| B: 规则约束路径（推荐） | 层级约束 + 关系白名单 |
| C: LLM 生成路径 | LLM 自主判断 |

**权衡**：
| 维度 | 盲搜 | 约束搜索 |
|:----|:----:|:--------:|
| 计算成本 | ❌ 高 | ✅ 低 |
| 噪声 | ❌ 高 | ✅ 低 |
| 医学合理性 | ❌ 不稳定 | ✅ 可控 |

**关键优化点——你当前设计缺了一个关键机制："路径语义合法性约束"**。

建议增加 ALLOWED_PATHS 定义：

```python
ALLOWED_PATHS = [
    (L0, L1, "causes"),
    (L1, L2, "causes"),
    (L2, L3, "causes"),
    (L3, L5, "treated_by")
]
```

否则你会出现"弯腰 → 椎间盘 → 手术 → 放射痛"这种结构正确但医学错误的路径。

---

### 决策 4：实体抽取策略（医学 NER 是否值得复杂化）

**推荐方案**：**规则 + 小模型（GLiNER）混合，但以规则为主**

| 维度 | GLiNER | 规则 |
|:----|:-----:|:----:|
| recall | ✅ 高 | ❌ 低 |
| precision | ❌ 不稳定 | ✅ 高 |
| 可控性 | ❌ 低 | ✅ 高 |

**为什么"简单反而更好"**：你的输入不是开放语料，而是高度结构化的临床描述。"放射痛"、"直腿抬高试验"、"L4/L5"——这些是有限词表问题，不是 NLP 问题。

---

### 决策 5：LLM 的角色（推理者 vs 解释器）

**推荐方案**：**LLM = 解释器，而不是推理核心**

| 维度 | LLM推理 | KG推理 |
|:----|:-------:|:------:|
| 可审计性 | ❌ 无 | ✅ 强 |
| 稳定性 | ❌ 波动 | ✅ 确定性 |
| 医疗安全性 | ❌ 风险高 | ✅ 可控 |

**核心原则**：路径 = ground truth，LLM = narrator。

你需要避免"LLM 看了一堆路径，然后自己重新推理一遍"，而应该强制"你只能基于以下路径生成结论，不得引入新因果关系"。

---

### 决策 6：证据融合策略（Graph vs Chunk 如何融合）

**推荐方案**：**Graph 决定"结构"，Chunk 提供"证据句子"**

**常见错误（你当前设计的潜在风险）**：
很多人会做 "Graph + Chunk → 拼一起 → 丢给 LLM"，这会导致 LLM 忽略 graph，或 graph 被 chunk 覆盖。

**推荐结构**：
```
[结构推理路径]
A → B → C → D

[证据支持]
- 文献1：支持 B→C
- 文献2：支持 C→D
```

**本质**：Graph = skeleton，RAG = muscle。

## 任务 B：风险与盲区

### 1. 你遗漏的关键问题

**a) "错误但合理路径"问题（最大风险）**
你的系统不会产生"胡说八道"，但会产生"看起来对，但其实错"。原因：图谱是人工构建的，关系缺少上下文约束。

**解决**：必须引入 relation direction constraints + layer transition rules。

**b) 知识覆盖不足（比 hallucination 更致命）**
MVP 最大问题不是错，而是"不知道"。Graph 的 recall 会非常低。

**c) SOAP 结构其实是 hardest part**
你低估了从结构化推理 → 临床语言表达的难度。这是 LLM 最容易"乱发挥"的地方。

### 2. 什么是过度工程？

**不必要（当前阶段）**：
- ❌ LangGraph
- ❌ 多 Agent 系统
- ❌ 自动文献抽取
- ❌ 在线更新 KG
- ❌ Neo4j

**必须做（否则系统没有说服力）**：
- ✅ 路径约束机制
- ✅ 证据绑定（edge → source）
- ✅ 输出可追溯（path id）

### 3. 面试官会问什么

**Q1：为什么不用纯 RAG？**
→ 你必须回答：RAG 无法提供因果结构，只能提供语义相似性。

**Q2：你的系统有什么局限？**
→ 你必须主动说：覆盖有限（small KG）、不具备概率推理、不适用于复杂共病、依赖人工知识建模。

**Q3：为什么不用知识图谱自动构建？**
→ 潜台词是"你这个系统 scale 不起来"。正确回答：MVP prioritizes epistemic clarity over scale。

## 任务 C：MVP 优先级（真实可跑路径）

### 必做（第一阶段）
1. **手工 KG（20节点 + 30边）** → 这是系统"灵魂"
2. **路径检索（带约束）** → 核心 differentiation
3. **Prompt 模板（强约束）** → 防止 LLM 乱来
4. **单 case 演示（Notebook）** → 比 100 个功能更重要

### 可以延后
- ❌ GLiNER（先用规则）
- ❌ FAISS（先用 keyword match 都可以）
- ❌ 多病种扩展
- ❌ 前端

### 可以砍掉（如果时间不够）
- 时间衰减机制 ❌
- 复杂关系类型 ❌
- 自动化 pipeline ❌

### 最关键一句总结

> 你的系统成败不在于"有没有用 FAISS / LLM / GLiNER"，而在于你是否成功把"医学推理结构"变成了一个可执行系统。
