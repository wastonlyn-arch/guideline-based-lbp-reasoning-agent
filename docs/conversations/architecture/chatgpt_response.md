作为你的 AI 架构师，我非常欣赏这个项目的立意。对于想要进入医疗 AI 领域的候选人来说，**"可解释性（Interpretability）"和"可审计性（Auditability）"**是绝对的护城河。大多数求职者只会拿开源数据集微调一个模型，而你选择了一条更具工程深度、也更契合真实临床痛点的路径——大模型（系统 1 的直觉）+ 分层图谱（系统 2 的逻辑）。

基于你的设计文档和现有资产，我将为你进行深度的架构剖析，不吹捧，直击核心痛点。

## 任务 A：独立识别关键决策（5个核心）

基于 8GB 内存、无 GPU、个人作品集这三个强约束，我认为以下 5 个决策将决定这个 MVP 的成败：

### 决策 1：图谱路径的计算层（Graph Computation Layer）

**你的原设计**：SQLite 递归 CTE 查询路径。

**我的推荐方案**：**SQLite（持久化存储） + NetworkX（内存计算）**。

**备选对比**：
| 方案 | 说明 | 评估 |
|------|------|------|
| A: 纯 SQLite 递归 CTE | 无额外依赖 | SQL 写复杂图遍历（特别是带有"层级必须 L0->L1->L2"这种逻辑约束、且要求返回完整路径而不仅仅是终点）极其痛苦且难以调试 |
| B: 引入 Neo4j | 图查询极强（Cypher 语言） | 过重，吃内存，部署麻烦，违背 MVP 原则 |
| **C: SQLite + NetworkX（推荐）** | 持久化 + 内存计算 | ≤2000 节点可毫秒级加载到内存，NetworkX 优雅处理最短路径、带权子图提取、拓扑排序 |

**为什么推荐 NetworkX**：对于 ≤2000 个节点的医学图谱，完全可以毫秒级加载到内存中。Python 的 NetworkX 库可以极其优雅地处理最短路径、带权子图提取、甚至是拓扑排序。面试时，展示你用 Python 优雅控制图算法，比写一坨难以维护的 SQL CTE 更加分。

---

### 决策 2：实体抽取引擎（Input Bottleneck）

**你的原设计**：GLiNER（零样本 NER）或 spaCy 规则。

**我的推荐方案**：**基于现有 extractor.py 的 LLM 结构化抽取（DeepSeek API） + 本地字典精确匹配（Fallback）**。

**备选对比**：
| 方案 | 说明 | 评估 |
|------|------|------|
| A: GLiNER | 轻量级本地模型 | 8GB 内存跑即使是小参数模型也会吃力，且对于中文特定的医学缩写和俗称（如"直腿抬高阳性"），未经微调的零样本表现通常很差 |
| B: spaCy/规则字典 | 极快 | 泛化能力为零，无法处理长难句 |
| **C: LLM API（推荐）** | 复用已有 extractor.py | 利用 DeepSeek API 强大的 Instruction Following 能力强制输出 JSON，最具性价比 |

**为什么推荐 LLM API**：既然你已经封装了 extractor.py 和 DeepSeek API，利用其强大的 Instruction Following 能力强制输出 JSON（包含症状、体征），是最具性价比的。你的作品集核心亮点是"推理链路"，不要在"实体抽取"这种前置任务上耗费太多本地算力，把本地算力留给向量检索。

---

### 决策 3：混合检索的编排策略（RAG vs. Graph-RAG）

**你的原设计**：图谱路径查询 和 向量检索（FAISS）平行运行，然后拼接 Prompt。

**我的推荐方案**：**图谱引导的向量检索（Graph-Steered Retrieval），即串行运行**。

**为什么推荐**：平行检索容易产生上下文割裂。更优雅的做法是：实体抽取 -> 在图谱中找到 L0 到 L5 的核心路径节点 -> 将这些节点名称作为查询词（Query），去 FAISS 中捞取强相关的文献证据（Chunks）。这样，你的图谱充当了"大纲"，向量库充当了"血肉"，这就形成了一个真正连贯的 Graph-RAG 故事，面试官会非常喜欢这个设计。

---

### 决策 4：分层约束的工程实现（Layer Enforcement）

**我的推荐方案**：在数据库/内存中强类型化定义 L0 到 L6，并在查询路径时做拓扑校验。

**为什么推荐**：这是你项目名字里的"分层（Hierarchical）"。你必须在代码层面体现出：患者主诉的"疼痛"被映射到 L3_Symptom，检查出的"直腿抬高"映射到 L4_Sign。当 Agent 推理"为什么"时，代码强制要求必须溯源经过 L2_Pathology（组织）和 L1_Biomechanics（力学）。这体现了深刻的医疗领域认知（Domain Knowledge）。

---

### 决策 5：对 LLM 的防御性编程（Anti-Hallucination Guardrails）

**我的推荐方案**：在 Prompt 中强制要求 LLM 使用引用标记，并通过后处理正则校验。

**为什么推荐**：医疗 AI 最怕幻觉。你的 Prompt 必须设计成这样："每提出一个评估（A）或计划（P），必须在句末标注来源，如 [Graph-Edge-12] 或 [Doc-Chunk-5]。" 在 LLM 返回后，用正则提取这些标记，去校验它们是否真的在你刚才查出的路径和列表中。如果有伪造标记，直接标记警告。这是"可审计性"的最佳体现。

## 任务 B：风险、盲区与面试防御

### 1. 你可能遗漏了什么（Blind Spots）？

**a) 否定语义的灾难**
患者说"没有下肢放射痛"。标准的 NER 会把"下肢放射痛"抽出来。如果不做上下文否定检测（Negation Detection），你的图谱会顺着"放射痛"查出神经根受压，导致误诊。

**缓解方案**：在实体抽取 Prompt 中必须要求 LLM 输出 `{"entity": "下肢放射痛", "present": false}`。

**b) 定量数据的丢失**
"直腿抬高 40°"和"直腿抬高 80°"在图谱里可能都指向同一个节点，但临床意义截然不同。你现有的概念节点设计难以承载定量值。

**缓解方案**：MVP 阶段可以在 Prompt 中用原始文本（S 和 O）来兜底，要求大模型结合原始文本中的定量数据来解释图谱中的定性关系。

### 2. 工程复杂度：什么是过度设计？

**过度设计（MVP阶段应直接砍掉）**：
- ❌ LangGraph 多 Agent 编排
- ❌ 动态从 PubMed 抓取新知识增量更新

**必要设计（必须保留）**：
- ✅ 别名归一化（Aliases 表）。临床口语化太严重（"腿麻" = "下肢放射痛"），如果没有别名映射，图谱就是瞎子
- ✅ 记录完整的日志（Logging），证明你有生产级系统的开发习惯

### 3. 面试官的火力点（Interview Prep）

**Q: "如果数据量大到 100 万节点，你的系统还能跑吗？"**
A: 现在的 SQLite+NetworkX 是应对 MVP 8GB 内存环境的妥协。如果是百万节点，我会将后端迁移到 Neo4j，利用 GraphSAGE 这种图神经网络来做图嵌入，甚至直接用 NebulaGraph 这种分布式图数据库。

**Q: "医学指南总是在变的，甚至存在矛盾，你的图谱怎么处理冲突？"**
A: 这就是为什么我在 edges 表设计了 source_ref 和 confidence 字段。图谱不产生真理，只映射证据。当存在矛盾（例如 A 指南推荐，B 指南禁忌），我在图遍历时会提取两条平行的冲突边，一并交给大模型（系统 1）去结合患者特征进行裁决。

**Q: "为什么不直接把医学指南丢进大模型做长文本推理，非要建个图？"**
A: 1. 黑盒问题：无法追溯中间推理链条；2. 跨层级缺失：大模型擅长模式匹配，但不擅长严密的"力学->组织->神经"多步因果逻辑演绎。图谱是充当大模型推理时的强制导航轨（Guardrails）。

## 任务 C：优先级建议（如何用最少代码出成果）

基于现有的 kg_system 资产，为了让你下周就能拿着 Demo 去面试，建议按以下顺序开发：

### 阶段 1：静态图谱 + 白盒推理（0 耗时前置准备）

**行动**：写死 1 个测试用例（比如腰椎间盘突出），在 SQLite 里手工插入 20 个核心节点、30 条边和 5 个 chunks。

**目标**：不写任何前端和抽取逻辑，直接写一个 Python 脚本，输入写死的"患者主诉实体"，验证 NetworkX 路径提取 -> FAISS 检索 -> 组装 Prompt -> LLM 生成带有 [Ref] 的 SOAP 报告 这一条链路。

### 阶段 2：输入端打通（复用你的 Extractor）

**行动**：把现有的 extractor.py 改装一下，限定它只能抽取症状和体征，并做好别名映射。

**目标**：现在你可以输入任意自然语言的主诉，系统能自动跑通了。

### 阶段 3：可视化审计（高光时刻）

**行动**：复用现有的 visualizer.py，不要画整个大图，而是动态生成当前患者专属的推理子图（高亮命中的路径 L0->L5），并随 SOAP 报告一起输出。

**目标**：视觉冲击力。面试时直接打开这个 HTML，指着高亮的红线说："这就是大模型的思考过程。"
