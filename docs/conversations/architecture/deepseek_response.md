## 任务 A：关键架构决策（7 个）

### 决策 1：知识图谱存储与查询引擎

**推荐方案**：A（SQLite + 递归 CTE）

| 方案 | 说明 |
|------|------|
| A：SQLite + 递归 CTE | 当前设计，零部署，路径查询用递归 SQL |
| B：Neo4j + Cypher | 专业图数据库，表达力强，但需额外服务 |
| C：NetworkX 内存图 | 纯内存，路径算法丰富，但无法持久化 |

**理由**：
- 你的现有资产中已有完整的 SQLite KG 实现（nodes/edges 表 + CRUD），稍改 schema 即可复用。
- 对于 MVP ≤2000 节点，递归 CTE 足以处理深度 ≤3 的路径，性能不是瓶颈。
- 零部署，面试官可以直接 git clone → python init_data.py 跑起来，最大程度降低环境摩擦——这对作品集至关重要。
- Neo4j 会让面试官额外安装服务，增加演示失败风险。

**权衡**：如果未来需要更复杂的图算法（PageRank、社区检测），SQLite 会吃力。但 MVP 不需要，且 NetworkX 可随时作为补充引擎，从 SQLite 读数据进内存计算，再写回。

### 决策 2：实体抽取方案

**推荐方案**：B（规则 + 词典）为主，A 作为增强

| 方案 | 说明 |
|------|------|
| A：GLiNER 零样本 NER | 文档推荐，本地运行，需加载模型 |
| B：spaCy 规则 + 词典 | 用领域词典匹配症状/动作，精准可控 |
| C：LLM 抽取（API） | 调用 DeepSeek 抽取实体，最省力但不稳定 |

**理由**：
- 你的领域极端狭窄（下腰痛/卒中康复），实体种类有限（症状、动作、解剖位、检查）。一个精心维护的词典（50-100 个词）配合 spaCy 的 Matcher 就能覆盖 90% 的场景。
- 规则抽取完全可解释、可调试、零延迟，符合"可审计"哲学。
- GLiNER 可作为兜底，但 MVP 阶段它对"直腿抬高试验"这类复合词效果可能不稳定，调试成本高。
- LLM 抽取会把核心逻辑变成黑箱，违背项目初衷。

**权衡**：词典需要手工维护，但这正是你展示领域建模能力的窗口。可以随时扩充，面试时清晰解释每个规则的来源。

---

### 决策 3：路径检索的深度与策略

**推荐方案**：B（关系过滤 + 方向感知）

| 方案 | 说明 |
|------|------|
| A：固定深度 3 的递归 CTE，返回所有路径 | 简单，但可能路径过多 |
| B：带关系过滤的递归，只抓取因果链 | 更精准，但需枚举关系类型 |
| C：加权路径搜索（关系 confidence × 路径长度） | 排序用，但 MVP 过度 |

**理由**：
- 你的关系类型已经明确定义（causes, treats, contraindicated_for 等），不同任务需要不同的关系子集。
- 如果返回所有关系类型的路径，会给 LLM 噪音（比如同时返回 is_a 和 contraindicated_for），反而不利于生成。
- 实现很简单：在递归 CTE 中加入 WHERE relation IN (...)。

**权衡**：需要根据不同意图预定义关系模板，但随着图谱增长，这是必要且优雅的控制。

---

### 决策 4：向量检索与图谱的融合方式

**推荐方案**：C（硬关联） + B（过滤检索）

| 方案 | 说明 |
|------|------|
| A：并行检索，拼接结果 | 图谱路径 + 向量 chunks 分别输入 LLM |
| B：图谱节点作为向量检索的过滤器 | 先用图谱找到相关概念，再在这些概念的 chunks 中检索 |
| C：硬关联（node_id） | chunks 表已有 node_id，直接 JOIN |

**理由**：
- 你的 chunks 表设计已包含 node_id，这是最自然的桥梁：图谱定位到相关节点 → 取这些节点的 chunks 作为证据 → 向量检索仅在这些 chunks 中做语义重排。
- 这比"全库向量搜索"更聚焦，避免检索到与当前患者无关的文献片段，同时保留了语义匹配的灵活性（比如患者说"腿麻"，能匹配到"放射痛"的 chunk）。
- 已有资产中的 VectorStore 可直接复用，只需限定检索范围。

**权衡**：要求 chunks 准确关联 node_id，这需要你在数据准备阶段做好映射。但工作量不大（每个 chunk 属于一两个概念）。

---

### 决策 5：LLM 的提示词策略与输出约束

**推荐方案**：B（JSON 输出） + 代码渲染 SOAP

| 方案 | 说明 |
|------|------|
| A：自由文本生成 SOAP | 靠 prompt 指导格式，但可能不稳定 |
| B：结构化 JSON 输出 + 后解析 | 强制 LLM 输出 JSON，解析后渲染 |
| C：分步生成（S → O → A → P） | 多次调用 LLM，分段生成，更可控 |

**理由**：
- DeepSeek API 支持 response_format: { type: "json_object" }，能稳定输出 JSON。
- 你将证据来源（图谱路径、文献 chunk）作为 JSON 字段附在每条结论后面，这使输出可直接被代码解析和展示，并且是可验证的。
- 分步生成会增加延迟和 token 消耗，对 MVP 不友好。

**权衡**：JSON 模式可能限制 LLM 的表达灵活性，但医疗报告格式本就固定，正好匹配。

---

### 决策 6：证据可信度建模

**推荐方案**：B（confidence + source_ref）

| 方案 | 说明 |
|------|------|
| A：仅保留文字引用 | 给出指南名或教材章节名 |
| B：边上的 confidence 分数 + 来源类型 | 设计文档中 edges 表已有 confidence 和 source_ref |
| C：引用链追溯（节点→chunk→文献） | 全链路，但工程量大 |

**理由**：
- 你的 edges 表设计已经包含了这两个字段，直接使用。设置 confidence 时，你的专家知识是最高分（0.9-1.0），指南推演次之，类比推理更低。
- 在 SOAP 报告中，每一条推理都带上来源和置信度，这本身就是"可审计"的强大体现，面试时会是亮点。
- C 的全链路追踪可以作为"未来扩展"提及，但不是 MVP。

**权衡**：confidence 分数主观性强，面试时你需要能解释打分依据（这正是展示你领域知识的时候）。

### 决策 7：整体工作流编排方式

**推荐方案**：A（函数链），用 Notebook 演示

| 方案 | 说明 |
|------|------|
| A：函数调用链（当前设计） | 硬编码顺序：抽取→检索→生成 |
| B：LangGraph 状态图 | 支持条件分支、重试、多 Agent |
| C：纯脚本，无编排 | 在 Jupyter Notebook 中按 cell 执行 |

**理由**：
- 你的目标不是构建一个生产级的 agent 框架，而是展示推理流程。在一个 Notebook 中将 extract_entities → retrieve_paths → retrieve_chunks → generate_report 逐步执行，并打印中间结果，是最直观的演示方式。
- LangGraph 会增加代码量和学习曲线，且会掩盖核心的领域建模亮点。它更适合你未来扩展时加入多 Agent，但在 MVP 中属于过度设计。

**权衡**：函数链缺少灵活的错误处理，但你可以复用已有的熔断器，对 LLM 调用部分做保护。

## 任务 B：风险与盲区

### 1. 你可能遗漏了什么？

**a) 实体消歧和标准化**
你的系统可能遇到"下腰痛"、"腰痛"、"腰背痛"等多个表述，如果直接作为节点 name 存储，会造成冗余和路径断裂。虽已有 aliases 表资产，但设计文档未明确提及如何使用。强烈建议在 nodes 表增加 canonical_name 并配合 aliases 解析。

**b) 路径循环的处理**
递归 CTE 如果没有正确终止，可能陷入无限循环（例如 is_a 自循环或双向边）。需要在 CTE 中加 WHERE depth < max_depth 并使用 UNION 去重。

**c) 冷启动问题**
系统依赖手工录入的 20-30 个节点和 30-50 条边。如果图谱覆盖面太窄，演示时可能找不到任何路径。需要提前设计好 2-3 个"黄金场景"（如腰椎间盘突出的标准推理链），确保这些路径一定存在。

**d) 向量检索的噪音**
现有 chunk 数据可能包含大段冗长文本，导致检索结果语义稀释。你的 Chunker 资产虽然可用，但需要针对医学段落微调（如按"机制-表现-干预"分割）。

**e) 法律与免责声明**
即使是一个作品集，在 README 和演示中也需要明确标注"原型系统，不可用于临床决策"。这能展示你的专业素养和对医疗 AI 伦理的认知。

### 2. "非生产系统"的复杂度取舍

**必须保留的复杂度**：
- 数据库事务和一致性（数据准确才能推理正确）
- 明确的日志（方便演示时展示流程和调试）
- 输入验证（防止空文本导致崩溃）

**可以省略的复杂度**：
- 身份认证、授权、加密存储（在本地跑不需要）
- 高并发处理和异步队列
- 自动化测试套件（手动跑通核心场景即可）
- CI/CD 管道

**判断标准**：如果一个组件的作用是"防止系统在真实用户压力下崩溃"或"保护患者隐私"，那在作品集中它不是必须的。但如果它能"展示你的工程品味或领域思考"（如日志格式、错误处理、配置分离），就值得花时间。

### 3. 面试官可能追问的 X

**Q: "为什么不用 Neo4j？"**
→ 解释作品集的零摩擦要求，但补充说可随时迁移，因为 Cypher 比 CTE 更适于复杂模式匹配。

**Q: "如何确保知识图谱的时效性？"**
→ 承认当前是静态快照，未来可通过自动文献抽取更新，并提及现有 chunk 的版本标记。

**Q: "你的系统如果遇到未见过的症状怎么办？"**
→ 说明会退回到纯向量检索，并提示"超出知识范围"，同时记录未覆盖的实体用于扩展。

**Q: "路径置信度如何校准？"**
→ 坦诚是专家经验，可引入贝叶斯更新或文献证据等级（如 GRADE），但 MVP 阶段未实现。

**Q: "为什么不直接用 LLM 端到端推理？"**
→ 这正是核心哲学：可解释、可审计，避免幻觉，符号化路径提供了坚实的骨架。

## 任务 C：最小可演示子集的优先级

### 第一阶段：骨架先行（第1-2天）

**数据库 schema 创建与初始化**
直接将你设计的 nodes/edges/chunks 表建好，并插入为"黄金场景"准备的 10 个节点、15 条边（覆盖腰椎间盘突出的因果链和治疗链）。复用你已有的 entities/aliases 结构，整合 canonical_name。

**实体抽取（规则版）**
用 spaCy 的 PhraseMatcher 加载一个 YAML 词典，包含 20 个症状/动作词。实现 extract_entities(text)。

### 第二阶段：核心推理（第3-4天）

**图谱路径查询**
实现 retrieve_graph_paths(entity, relation_filter=["causes","associated_with"])，用递归 CTE，并加入深度和循环控制。返回标准化路径列表（如 ["长期坐姿不良" -> causes -> "椎间盘退变" -> causes -> "纤维环破裂"]）。

**向量检索集成**
插入 5-8 个 chunks，关联到关键节点。实现 retrieve_chunks(entities, query_text, top_k=3)，限定在给定实体的 chunks 内用 FAISS 搜索。复用已有 embedder 和 VectorStore。

**提示词组装与调用**
构建一个 prompt 模板，注入患者信息、图谱路径、chunks 上下文，要求 LLM 输出 JSON 格式的 SOAP。调用 DeepSeek API。

### 第三阶段：串联与演示（第5-6天）

**orchestrator.py**
将上述步骤串联为一个函数 medical_agent(patient_text)，并加入简单的日志输出每一步的耗时和中间结果。

**example.ipynb**
用 Notebook 展示至少 2 个场景：一个是标准"腰椎间盘突出"推理，另一个是故意给一个图谱无法覆盖的症状（如"肩痛"），展示 fallback 行为。

**README.md**
写清楚如何安装依赖（requirements.txt 直接从你的模板复制）、如何运行、以及架构图。包含免责声明。

### 可暂时推迟的：
- GLiNER 集成（词典够用）
- LangGraph 编排（函数链即可）
- 多病种扩展（专注下腰痛）
- 前端 UI（Notebook 演示足够冲击力）
- 置信度建模的精细化（先给固定分数）

## 规则增强建议（Rules 审查与增强）

### 新增规则

| 规则 | 内容 | 对应风险 |
|------|------|---------|
| **规则 8：实体标准化优先级** | 所有入 KG 实体经过 canonical_name 映射，原始字符串不入库 | Gagle/EAGLE 问题 |
| **规则 9：路径查询安全约束** | 递归 CTE 必须有 max_depth ≤ 3 + 去重 + 明确方向 | CTE 失控锁死 |
| **规则 10：检索边界控制** | 图谱筛选 entity set → 语义重排，禁止全库搜索 | 检索噪音 |
| **规则 11：LLM 输出强约束** | JSON 模式 + schema 校验 + retry/fallback | 解析失败 |
| **规则 12：证据绑定强制规则** | 结论必须绑定 source_ref + 类型 + confidence | 可审计系统 |
| **规则 13：冷启动保障** | 预置 ≥2 个黄金推理路径 | 空图谱演示 |
| **规则 14：Fallback 可观测** | 明确输出 fallback 状态 + 日志记录未覆盖实体 | 静默失败 |
| **规则 15：输出免责声明** | 所有输出带免责声明 | 法律风险 |
| **规则 16：最小闭环定义** | 端到端管线每步有日志、可单独运行、可验证 | 交付标准 |
| **规则 17：演示优先级原则** | 先跑通完整流程 → 再解释每一步 → 再优化 | 开发秩序 |

### 最终规则集收敛

| 类别 | 规则 |
|------|------|
| **核心控制类（7条）** | 规则1(开发分级)、规则4(数据安全)、规则5(可回滚)、规则8(实体标准化)、规则9(路径安全)、规则10(检索边界)、规则12(证据绑定) |
| **工程质量类（5条）** | 规则2(代码规范)、规则3(环境依赖)、规则6(模块交付)、规则11(LLM输出)、规则14(fallback可观测) |
| **演示保障类（5条）** | 规则7(模块修改)、规则13(冷启动)、规则15(免责)、规则16(最小闭环)、规则17(优先级) |
