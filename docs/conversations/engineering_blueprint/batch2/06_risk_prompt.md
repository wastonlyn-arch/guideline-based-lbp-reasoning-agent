## 项目背景

- 项目名称: Clinical Reasoning Agent — MCRM 物理治疗临床推理教学辅助
- 当前规模: 单人开发（即你），分散时间投入（每周 10-15 小时）
- 技术栈: Python 3.11、SQLite + NetworkX、FAISS、sentence-transformers、DeepSeek API
- 核心依赖: LLM API（外部不可控）、临床指南 PDF 数据（版权许可）、知识图谱手工构建
- 当前进度: v0.3，核心功能可用但未完整测试

## 任务要求

请输出一个完整的风险矩阵（Risk Matrix），覆盖以下 4 大类风险：

### 1. 技术风险
每个风险条目格式：
| 风险 ID | 风险描述 | 概率 (H/M/L) | 影响 (H/M/L) | 风险等级 | 触发条件 | 应对策略 | 应急计划 |

至少列出以下风险：
- LLM API 不可用 / 限流 / 返回格式异常
- FAISS 索引与 SQLite 图谱数据不一致
- 实体抽取的词典不完整导致召回率过低
- NetworkX 路径检索在 5000+ 节点时性能崩溃
- sentence-transformers 模型加载内存溢出
- 多模型交叉验证的 convergence 失败

### 2. 数据风险
- 临床指南未获版权授权
- PDF 结构混乱导致自动抽取失败
- 中文术语映射不准（如"LBP" = "下腰痛"但多个同义词）
- 图谱中的专家标注错误传播

### 3. 进度风险
- 单人开发被其他工作打断（长期搁置）
- 知识图谱构建周期远超预期（900 页指南 2000 节点已经耗时巨大）
- 测试覆盖率迟迟无法提升

### 4. 质量风险
- LLM 生成 SOAP 的医学准确性无法保证
- 推理路径中遗漏关键鉴别诊断
- 置信度评分不准确导致误导用户

### 5. 风险矩阵汇总
请将所有风险汇总到一个风险等级热图（Heatmap）中，按 概率×影响 分类到 5 个优先级等级：

| 优先级 | 概率×影响 | 行动要求 |
|--------|----------|---------|
| P0 (Critical) | H×H | 必须立即处理，影响项目存亡 |
| P1 (High) | H×M / M×H | 必须制定应对策略，在本 sprint 解决 |
| P2 (Medium) | M×M / L×H | 需要持续监控，必要时响应 |
| P3 (Low) | L×M / M×L | 只需要记录，暂不行动 |
| P4 (Monitor) | L×L | 仅有文档记录 |

每个风险都要给出具体的触发条件和应对策略，不能泛泛而谈。

## 审查指南

请从以下角度逐条审查：

1. **遗漏风险**：有哪些重要风险被忽略了？
   - 例如：单人开发的健康风险（burnout）？
   - 例如：Streamlit future 前端的交付风险？
   - 例如：依赖的第三方包不再维护？
   - 例如：知识图谱构建工具链的维护负担？
2. **风险等级误判**：哪些风险的概率/影响被高估或低估了？
   - LLM API 不可用的真实概率
   - 图谱构建周期延期的真实概率（几乎是必然的）
3. **应对策略空洞**：策略是否可执行？
   - "加强测试" → 空洞
   - "每次 git commit 自动跑 pytest --cov=src --cov-fail-under=80" → 可执行
4. **风险之间的依赖**：某个风险的发生是否会级联放大其他风险？
   - 例如：知识图谱延期 → 测试没数据写 → 覆盖率上不去 → 质量风险上升
5. **MVP 阶段特殊性**：对于 MVP 演示系统，哪些风险可以接受（如 QA 不完整），哪些不可接受（如演示时 API 挂掉）？

请列出所有问题，每个问题给出：
- 严重性（High/Medium/Low）
- 具体说明
- 你的修改建议
