## Clinical Reasoning Agent — MCRM 物理治疗临床推理教学辅助

### 1. 当前架构的性能瓶颈分析

基于您的技术栈和图谱规模，以下是各模块的性能预估和潜在瓶颈：

*   **NetworkX 全图 BFS/DFS 遍历 (内存图)**
    *   **复杂度:** BFS/DFS 的时间复杂度为 O(V+E)，其中 V 是节点数，E 是边数。对于知识图谱，通常是稀疏图，E 接近 O(V)。
    *   **预估耗时 (基于典型 CPU 性能):**
        *   **2000 节点 (约 5000-10000 边):** 路径检索 (MAX_LENGTH=4, MAX_PATHS=10) 预计在 **50-200 ms**。
        *   **5000 节点 (约 15000-25000 边):** 路径检索预计在 **150-500 ms**。
        *   **15000 节点 (约 45000-75000 边):** 路径检索预计在 **500-1500 ms**。
    *   **瓶颈分析:**
        *   **内存:** NetworkX 将整个图加载到内存。15000 节点，每个节点和边存储额外属性，可能占用数百 MB 到数 GB 内存。对于单个应用实例尚可，但多并发实例可能导致内存压力。
        *   **单线程:** NetworkX 操作是单线程的，高并发下会成为瓶颈。
        *   **路径组合爆炸:** 即使有 MAX_PATHS 和 MAX_LENGTH 限制，如果起始节点过多或图结构高度互联，仍可能导致计算量大增。

*   **FAISS 检索 (IndexFlatIP, 嵌入维度 384)**
    *   **复杂度:** `IndexFlatIP` 是暴力搜索，时间复杂度为 O(N*D)，其中 N 是向量数量，D 是维度。
    *   **预估耗时 (基于典型 CPU 性能):**
        *   **2000 向量 (384 维):** 余弦相似度 top-k 检索预计在 **5-20 ms**。
        *   **5000 向量 (384 维):** 检索预计在 **15-50 ms**。
        *   **15000 向量 (384 维):** 检索预计在 **50-200 ms**。
    *   **瓶颈分析:**
        *   **可扩展性:** `IndexFlatIP` 在向量数量超过 100,000 甚至 50,000 时，性能会急剧下降。对于当前规模，性能良好。
        *   **内存:** 15000 * 384 * 4 字节 (float32) ≈ 23 MB，内存占用很小。

*   **SQLite 查询 (JOIN nodes/edges)**
    *   **预估表现:**
        *   **加载图谱到 NetworkX:** 这是主要使用场景。对于 5000+ 节点，如果表结构和索引设计合理 (例如，`nodes` 表主键，`edges` 表外键和索引)，全量加载可能需要 **1-5 秒**。
        *   **单次查询 (如获取节点属性):** 毫秒级。
    *   **瓶颈分析:**
        *   **I/O 限制:** SQLite 是文件系统上的数据库，I/O 性能受限于磁盘速度。
        *   **并发写入:** SQLite 对于并发写入支持有限，但在您的教学场景中，写入不频繁，主要瓶颈在于全量读取。
        *   **NetworkX 内存图的持久化:** SQLite 只是 NetworkX 内存图的持久化存储，实际查询性能瓶颈不在 SQLite 本身，而在于加载到 NetworkX 的耗时。

*   **LLM API 调用的延迟影响 (每次 2-5s)**
    *   **瓶颈分析:** 这是当前架构中**最显著**的瓶颈。任何涉及 LLM 调用的用户请求，其响应时间都将至少是 2-5 秒。
    *   **影响:** 严重影响用户体验，使得整个系统响应缓慢。必须通过缓存、异步处理或减少 LLM 调用次数来缓解。

### 2. 路径组合爆炸防护

您已设置 `MAX_PATHS=10` 和 `MAX_LENGTH=4`，这是非常好的起点。

*   **这个阈值是否足够防止爆炸？**
    *   **足够应对大部分情况。** `MAX_LENGTH=4` 意味着最多 4 跳，这限制了搜索深度。`MAX_PATHS=10` 限制了广度。对于一个平均度数在 5-10 的稀疏知识图谱，这个组合能有效控制搜索空间。
    *   **潜在风险:** 如果图谱中存在少数“超级节点” (度数极高)，或者用户输入映射到的起始节点本身就处于一个高度互联的区域，仍可能在达到 MAX_PATHS 前进行大量计算。但总体而言，这个阈值是合理的。

*   **如果用户输入模糊 (如"腰痛"映射到 10+ 起点节点)，如何限流？**
    *   **FAISS 结果筛选:**
        *   **Top-K 限制:** 在 FAISS 检索后，只取相似度最高的 `K` 个节点作为起始点 (例如 `K=3` 或 `K=5`)。这能有效减少后续图遍历的并行度。
        *   **相似度阈值:** 设置一个相似度阈值 (例如 0.75)。只有相似度高于此阈值的节点才被考虑为起始点。
    *   **用户澄清:** 如果通过 FAISS 筛选后，仍然有多个起始节点具有相似且较高的相关性，可以提示用户进行选择或澄清，例如：“‘腰痛’可能指向多个概念，您是指‘急性腰痛’、‘慢性腰痛’还是‘腰椎间盘突出’？”
    *   **优先级排序:** 如果有额外信息 (如节点类型、重要性)，可以对起始节点进行加权和排序。

*   **是否需要断路器模式 (超过 N ms 后降级)？**
    *   **强烈建议需要。** 尤其对于图遍历这种计算密集型且可能耗时的操作。
    *   **阈值建议:** 对于图遍历，可以设置一个相对宽松的阈值，例如 **1000-1500 ms** (不含 LLM 调用)。
    *   **降级策略:**
        *   **返回部分结果:** 如果在超时前找到了部分路径，返回这些路径。
        *   **提示用户:** 告知用户查询过于复杂，建议细化输入或尝试其他关键词。
        *   **简化查询:** 自动降低 `MAX_LENGTH` 或 `MAX_PATHS`，然后重新尝试。

*   **是否需要缓存常见查询的热路径？**
    *   **强烈建议需要。** 常见查询的热路径是缓存的理想候选者。
    *   **策略:**
        *   识别高频查询 (例如，通过日志分析)。
        *   将这些查询的完整路径结果 (节点序列、边序列) 缓存起来。
        *   缓存键可以包括起始节点 ID、目标节点 ID (如果指定)、MAX_LENGTH、MAX_PATHS 等参数的哈希值。

### 3. 图数据库迁移决策树

从 SQLite + NetworkX 迁移到专用图数据库 (Neo4j / FalkorDB) 是一个重要的决策，应基于实际性能瓶颈和功能需求。

*   **决策树:**

    1.  **节点数 > X → 评估**
        *   **阈值:** `节点数 > 50,000` 或 `图谱内存占用持续超过 80% 的可用 RAM` (对于 NetworkX 进程)。
        *   **评估点:** NetworkX 加载时间是否过长 (例如，启动或热加载超过 10 秒)？NetworkX 内存占用是否导致系统不稳定？

    2.  **图遍历查询延迟 > X ms (不含 LLM) → 评估**
        *   **阈值:** `核心图遍历查询 (如 BFS/DFS) 延迟持续超过 1000 ms` (在 MAX_LENGTH 和 MAX_PATHS 限制下)。
        *   **评估点:** NetworkX 是否在 CPU 利用率上达到瓶颈？是否需要更复杂的图算法 (如社区发现、中心性计算)？

    3.  **并发用户 > X → 评估**
        *   **阈值:** `并发活跃图查询用户数 > 5` (即同时有 5 个以上用户在进行图遍历操作)。
        *   **评估点:** NetworkX 的单线程特性是否导致请求排队，响应时间变慢？是否需要多个应用实例共享图谱状态？

    4.  **图谱更新频率和复杂性 → 评估 (额外触发条件)**
        *   **阈值:** `图谱需要频繁 (例如，每天多次) 实时更新` 或 `更新操作涉及复杂的事务和一致性要求`。
        *   **评估点:** SQLite + NetworkX 的更新流程 (更新 SQLite -> 重新加载 NetworkX) 是否过于繁琐或耗时？

    5.  **数据模型复杂性 → 评估 (额外触发条件)**
        *   **阈值:** `需要支持更丰富的节点/边属性模型` 或 `需要更灵活的查询语言 (如 Cypher)`。
        *   **评估点:** SQLite 的关系型模型是否已经无法高效表达图谱的复杂关系？

*   **总结:** 在 MVP 阶段，SQLite + NetworkX 足够。当上述任一阈值被触发时，应启动对 Neo4j 或 FalkorDB 的评估。Neo4j 社区版是一个很好的起点，FalkorDB (RedisGraph) 在内存性能上可能更优，但生态相对较小。

### 4. 抽象接口层设计

*   **是否需要从一开始设计 GraphBackend 接口让 NetworkX 和 Neo4j 可互换？**
    *   **强烈建议从一开始就设计。** 即使是 MVP，一个薄的抽象层也能带来巨大的收益。
    *   **接口示例 (伪代码):**
        ```python
        class GraphBackend:
            def add_node(self, node_id, properties): pass
            def add_edge(self, source_id, target_id, properties): pass
            def get_node(self, node_id): pass
            def get_neighbors(self, node_id): pass
            def find_paths(self, start_node_id, end_node_id=None, max_length=4, max_paths=10): pass
            # ... 其他核心图操作

        class NetworkXBackend(GraphBackend):
            def __init__(self, graph: nx.Graph): self._graph = graph
            # ... 实现 GraphBackend 接口

        # class Neo4jBackend(GraphBackend):
        #     def __init__(self, uri, auth): self._driver = GraphDatabase.driver(uri, auth)
        #     # ... 实现 GraphBackend 接口
        ```
*   **如果现在不设计，未来迁移成本有多大？**
    *   **迁移成本巨大。** 如果不设计抽象层，所有业务逻辑中直接调用 `networkx.Graph` 对象的方法 (例如 `graph.add_node()`, `graph.neighbors(node_id)`, `nx.bfs_edges()`) 的地方都需要重写。
    *   **具体成本预估:**
        *   **代码修改量:** 假设有 500-1000 行代码直接与 NetworkX 交互，迁移可能需要修改其中 80% 以上。
        *   **引入 Bug 风险:** 大规模重构必然引入新的 Bug，需要大量的回归测试。
        *   **时间成本:** 至少需要 **2-4 周** 的专门重构工作，外加数周的测试和验证。
    *   **结论:** 早期投入少量时间设计抽象层，可以避免未来数周甚至数月的重构成本和风险。

### 6. 缓存策略

*   **哪些数据适合缓存？**
    1.  **LLM 响应:** **优先级最高。** LLM 调用延迟高且成本高，相同输入通常返回相同或相似结果。
        *   **缓存键:** LLM 输入提示词 (prompt) 的哈希值。
    2.  **嵌入向量 (Embeddings):** `sentence-transformers` 生成的向量。一旦文本内容确定，其嵌入向量就是固定的。
        *   **缓存键:** 原始文本内容的哈希值或节点 ID。
    3.  **图谱查询结果 (热路径):** 频繁查询的特定路径或子图。
        *   **缓存键:** 起始节点 ID、目标节点 ID (如果指定)、MAX_LENGTH、MAX_PATHS 等参数的组合哈希值。
    4.  **FAISS 检索结果:** 对于特定用户查询文本，FAISS 返回的 Top-K 节点 ID 列表。
        *   **缓存键:** 用户查询文本的哈希值。

*   **缓存失效策略？**
    1.  **LLM 响应:**
        *   **TTL (Time-To-Live):** 设置一个相对较长的过期时间，例如 `24 小时` 到 `7 天`。
        *   **手动失效:** 如果 DeepSeek API 的模型更新导致输出行为显著变化，可以手动清空 LLM 缓存。
    2.  **嵌入向量:**
        *   **内容变更触发:** 当某个节点或边的**文本内容**发生变化时，对应的嵌入向量缓存失效。
        *   **图谱更新时:** 如果图谱更新涉及大量节点文本变更，可以考虑批量失效或重建。
    3.  **图谱查询结果 (热路径):**
        *   **图谱结构变更触发:** **最关键。** 只要图谱的**任何节点或边**被添加、删除或修改，所有依赖于图结构计算的路径缓存都应失效。
        *   **策略:** 可以采用版本号机制。每次图谱更新，递增一个全局图谱版本号。缓存键中包含这个版本号，当版本号不匹配时，缓存自动失效。或者，直接进行全局清空。
    4.  **FAISS 检索结果:**
        *   **FAISS 索引重建/更新时:** 当 FAISS 索引因为新增、删除或更新了节点嵌入向量而重建时，所有 FAISS 检索结果缓存都应失效。

*   **Redis vs 内存 LRU 缓存？**
    *   **MVP 阶段 (单应用实例): 内存 LRU 缓存 (例如 `functools.lru_cache`) 优先。**
        *   **优点:** 实现简单，无需额外部署和维护，访问速度极快。
        *   **缺点:** 缓存数据不持久化，应用重启后丢失；无法在多个应用实例之间共享。
        *   **适用场景:** LLM 响应、嵌入向量、热路径等。对于当前教学项目，单个 Python 进程足以承载，内存 LRU 缓存是最佳选择。
    *   **未来扩展 (多应用实例/持久化/大数据量): Redis。**
        *   **优点:** 支持分布式缓存，多个应用实例可共享；数据可持久化；支持更大的缓存容量；提供更丰富的缓存策略和数据结构。
        *   **缺点:** 引入外部依赖，增加部署和维护复杂性；网络延迟会略高于内存访问。
        *   **触发迁移条件:**
            *   当您需要部署多个应用实例来处理并发请求时。
            *   当缓存数据量超出单个应用进程的内存限制时。
            *   当缓存数据需要持久化，即使应用重启也不丢失时。

**总结:**
在 MVP 阶段，专注于利用 `functools.lru_cache` 实现内存缓存，特别是针对 LLM 响应和高频图谱查询。当项目进入生产或需要扩展时，再考虑引入 Redis。
