## 1. 分阶段交付路线图（P1-P4）

### 总体策略

采用分阶段交付路线图（P1-P4），每个阶段包含可验证的里程碑和审查检查点。在每个阶段执行前，增加审查检查点：审核数据准备时间估算是否合理、是否有 fallback 方案、里程碑是否可量化验证。

### P1 — 核心引擎 + 最小闭环（MVP 基线）

| 里程碑 | 交付物 | 验证标准 |
|:-------|:-------|:---------|
| M1.1 | 核心 Agent 框架搭建 | 主循环 `run()` 可启动、AgentState 流转正确 |
| M1.2 | LLM 客户端封装（流式/非流式） | 支持 async 流式对话、错误重试、超时处理 |
| M1.3 | 最小知识图谱（2-3 病种） | 可查询、路径推理、GraphBackend 接口可用 |
| M1.4 | 基础推理链路跑通 | 从用户输入 → 推理 → 报告生成完整闭环 |
| M1.5 | 单元测试覆盖 ≥ 60% | pytest 通过率 100% |

**审查检查点**：P1 结束时评审知识图谱数据质量、LLM 调用稳定性、测试覆盖率是否达标。

### P2 — 数据工程 + 知识图谱扩展

| 里程碑 | 交付物 | 验证标准 |
|:-------|:-------|:---------|
| M2.1 | 多源数据接入管道（电子书/论文/指南） | 非结构化文本解析流水线可运行 |
| M2.2 | 知识抽取流水线（规则 + LLM + 实体库） | 抽取准确率 ≥ 85%（人工抽检） |
| M2.3 | Schema Registry 实现 | 版本管理、动态更新、向后兼容 |
| M2.4 | 知识图谱扩展至 5+ 病种 | 含关联关系，支持跨病种查询 |
| M2.5 | 性能 profiling 完成 | NetworkX 延迟、embedding 生成延迟报告 |

**审查检查点**：数据工程时间估算是否合理？是否有 fallback 方案（如预处理数据缓存）？公共数据集是否已准备为平行轨道？

**风险处理**：若 NetworkX 实际延迟超出预估，立即启动图数据库迁移评估。

### P3 — 测试 + 性能 + CI

| 里程碑 | 交付物 | 验证标准 |
|:-------|:-------|:---------|
| M3.1 | 集成测试覆盖核心链路 | 含 Mock/Fake 外部依赖、SQLite 内存库 |
| M3.2 | E2E 测试覆盖完整业务流程 | 患者主诉 → SOAP 报告全链路 |
| M3.3 | GitHub Actions CI/CD 流水线 | 代码格式化 + 类型检查 + 测试 + 并发测试 |
| M3.4 | 性能基准测试 | P99 延迟 ≤ 5s、TP99 ≤ 8s |
| M3.5 | 缓存机制（LLM 响应缓存 + 热路径缓存） | 缓存命中率 ≥ 40%（估算） |

**审查检查点**：测试覆盖质量门控是否达标？性能瓶颈是否已识别并归档？

### P4 — 多模型集成 + 部署上线

| 里程碑 | 交付物 | 验证标准 |
|:-------|:-------|:---------|
| M4.1 | MultiLLM 分层自动触发 | Tier2+ 任务默认启用 light_multi 模式 |
| M4.2 | 成本预算检查 + 自动降级 | 预算超限时自动回退 single 模式 |
| M4.3 | Streamlit Cloud 演示部署 | 固定假数据可交互演示 |
| M4.4 | Conda 环境 + 文档清单 | `environment.yml` + 首次启动检查清单 |
| M4.5 | 预提交钩子 + 代码风格统一 | pre-commit hooks 配置完成 |

**审查检查点**：部署方案是否完善？多模型成本是否可控？文档是否覆盖故障排查？

## 3. 数据与知识图谱架构

### 多源数据接入

```
┌──────────────────────────────────────────┐
│              数据源                       │
│  ┌─────────┐ ┌─────────┐ ┌───────────┐  │
│  │ 医学指南  │ │ 诊疗论文  │ │ 临床路径   │  │
│  └────┬─────┘ └────┬─────┘ └─────┬─────┘  │
│       │            │              │        │
│  ┌────▼────────────▼──────────────▼─────┐  │
│  │      非结构化文本解析管道              │  │
│  │  (PDF 解析 + 章节分割 + 段落标注)      │  │
│  └────────────────┬──────────────────────┘  │
└───────────────────┼──────────────────────────┘
                    │
┌───────────────────▼──────────────────────────┐
│          知识抽取流水线                        │
│  ┌──────────┐ ┌──────────┐ ┌──────────────┐ │
│  │  规则引擎  │ │  字典匹配  │ │  LLM 辅助    │ │
│  │ (正则/句法)│ │ (医学术语)│ │ (实体/关系)  │ │
│  └──────────┘ └──────────┘ └──────┬───────┘ │
│                                      │        │
│  ┌──────────────────────────────────▼────┐    │
│  │          实体库校验 + 上下文消歧        │    │
│  └───────────────────────────────────────┘    │
└────────────────────────────────────────────────┘
```

**决策要点**：
- **规则 + 字典 + LLM + 实体库** 四层校验流水线，降低 LLM 幻觉风险
- Jina/UnstructuredIO 处理非结构化文本，后期按需引入 `neo4j`
- 抽象 `GraphBackend` 接口，初期使用 NetworkX，后期可按需切换至图数据库

### Schema Registry

- 独立 Schema 注册中心，支持版本控制和动态更新
- 节点、关系、属性均带版本标签
- 增量更新自动检测冲突并支持人工复核
- 版本管理自动触发代码生成或提示变更

### 知识图谱规模预估

| 病种数 | 预估节点数 | 预估关系数 | 存储需求 |
|:-------|:-----------|:-----------|:---------|
| 3 (P1) | 3,000-8,000 | 8,000-20,000 | ~50MB |
| 5 (P2) | 8,000-20,000 | 20,000-60,000 | ~150MB |
| 10+ (扩展) | 30,000-80,000 | 80,000-200,000 | ~500MB |

> NetworkX 在 10 万节点以内性能可接受（<200ms 查询），但需持续监控。

## 4. 测试策略

### 测试金字塔

```
         ╱╲
        ╱  ╲        E2E 测试
       ╱    ╲       (2-3 条关键链路)
      ╱──────╲
     ╱        ╲     集成测试
    ╱          ╲    (Mock/Fake 外部依赖)
   ╱────────────╲
  ╱              ╲  单元测试
 ╱                ╲ (各模块独立测试)
╱──────────────────╲
```

### 各模块最小可测单元

| 模块 | 单元测试 | Mock 策略 | 关键测试点 |
|:-----|:---------|:----------|:-----------|
| `LLMClient` | 流式/非流式调用 | Mock API 响应 | timeout、重试、降级 |
| `GraphBackend` | CRUD 操作 | Fake SQLite/NetworkX | 路径查询、版本冲突 |
| `AgentState` | 状态流转 | 无外部依赖 | 全部状态转换矩阵 |
| `ReasoningEngine` | 推理链路 | Mock LLM + Mock KG | 异常输入、边界条件 |
| `ReportGenerator` | 报告生成 | Mock 推理结果 | 格式校验、内容完整性 |

### 关键测试覆盖要求

1. **流式响应模拟** — 不仅测试非流式 Mock，还需测试流式 token 逐片推送上屏
2. **异常/错误路径** — LLM timeout、FAISS 无结果、KG 节点不存在、网络中断
3. **并发测试** — CI 工作流中基础测试通过后执行并发压力测试
4. **测试优先级** — P0 模块必须 100% 覆盖；P1 模块 ≥ 80%；P2 模块 ≥ 60%

### 测试数据管理

- 测试数据版本控制（`tests/fixtures/` 目录）
- 使用 in-memory SQLite 代替生产数据库进行集成测试
- 定期更新测试数据，保持与真实数据一致性

## 5. 性能基线

### 架构方案

```
┌─────────────────────────────────────────────┐
│            MVP 基础架构                       │
│                                              │
│  ┌──────────────┐    ┌──────────────────┐    │
│  │    SQLite     │    │    NetworkX       │    │
│  │ (持久化存储)   │◄──►│ (内存图结构)      │    │
│  └──────┬───────┘    └────────┬─────────┘    │
│         │                      │              │
│  ┌──────▼──────────────────────▼────────┐    │
│  │          GraphBackend 接口            │    │
│  │   (抽象层，可切换后端实现)             │    │
│  └────────────────┬─────────────────────┘    │
└───────────────────┼──────────────────────────┘
                    │
┌───────────────────▼──────────────────────────┐
│               缓存层                          │
│  ┌──────────────┐ ┌──────────────────────┐  │
│  │ LLM 响应缓存  │ │ 热路径查询缓存        │  │
│  │ (in-memory   │ │ (in-memory LRU)      │  │
│  │  LRU + 8KB)  │ │                      │  │
│  └──────────────┘ └──────────────────────┘  │
└──────────────────────────────────────────────┘
```

### 延迟目标

| 指标 | P1 | P2 优化后 | 边界 |
|:-----|:---|:----------|:-----|
| P50 端到端推理延迟 | ≤ 3s | ≤ 2s | 含 LLM 调用 |
| P99 推理延迟 | ≤ 5s | ≤ 3s | 含缓存 miss |
| TP99 吞吐 | ≤ 8s | ≤ 5s | 含并发 |
| KG 路径查询（P50） | ≤ 50ms | ≤ 20ms | 10万节点规模 |
| Embedding 生成 | ≤ 2s | ≤ 1.5s | 512 chunk |
| FAISS 检索（P50） | ≤ 30ms | ≤ 15ms | 5000 向量 |

### 路径组合爆炸防护

- `MAX_PATHS = 10`
- `MAX_LENGTH = 4`
- 断路器模式：连续 3 次 timeout → 自动熔断 30s

### 监控与 Profiling

- 各模块内置计时装饰器 `@timed()`
- 性能指标输出结构化 JSON 日志
- MVP 初期快速验证 NetworkX 实际延迟，若超出预估则启动图数据库迁移评估

## 6. 风险矩阵

| 编号 | 风险 | 概率 | 影响 | 等级 | 应对策略 | 依赖关系 |
|:-----|:-----|:-----|:-----|:-----|:---------|:---------|
| TR-01 | LLM API 故障 / 超时 | 中 | 高 | P0 | 本地缓存（32条） + 备用 API 列表 + 断路器熔断 | 无 |
| TR-02 | 知识图谱构建延期 | 高 | 高 | P0 | 并行准备公共数据集 + 简化 P1 病种范围 | 数据管道 |
| TR-03 | 单人开发健康风险（burnout） | 低 | 高 | P0 | 合理排班、定时提醒、预留外部支援通道 | 无 |
| TR-04 | 测试覆盖率不足 | 中 | 中 | P1 | 自动化 CI 质量门控 + 覆盖率报告 | CI 流水线 |
| TR-05 | Embedding 生成延迟超标 | 中 | 中 | P1 | 异步生成 + 缓存 + chunk 大小优化 | 数据管道 |
| TR-06 | NetworkX 性能瓶颈 | 中 | 中 | P1 | 提前 profiling + GraphBackend 抽象层 | 架构设计 |
| TR-07 | LLM 幻觉导致错误推理 | 高 | 高 | P1 | 多模型交叉验证 + 实体库校验 + human-in-loop | 知识抽取 |
| TR-08 | 多模型成本超支 | 中 | 中 | P2 | 成本预算检查 + 自动降级 + 仅 Tier2+ 启用 | MultiLLM |
| TR-09 | 前端框架版本兼容 | 低 | 低 | P2 | 版本锁 + 升级测试计划 | 无 |
| TR-10 | 文档滞后于代码 | 高 | 低 | P2 | 半自动文档更新 + 核心文档优先 | 无 |

### 风险应对优先级

1. **TR-01 / TR-02 / TR-03** — P0 级风险，必须在 P1 阶段制定具体应对策略并落地
2. **TR-04 / TR-05 / TR-06 / TR-07** — P1 级风险，在 P2/P3 阶段逐步解决
3. **TR-08 / TR-09 / TR-10** — P2 级，低优先级，持续监控

## 7. 多模型集成策略

### 分层自动触发架构

```
输入请求
    │
    ├── Tier0（简单查询）─────► 单一 cheap 模型
    │
    ├── Tier1（一般推理）─────► 单一 primary 模型
    │
    ├── Tier2（复杂推理）─────► light_multi（1 primary + 1 critic）
    │
    ├── Tier3（关键决策）─────► full_multi（2-3 模型交叉验证）
    │
    └── Tier4（紧急/高风险）──► 全模型 + human-in-loop
```

### 默认配置

| 参数 | 值 | 说明 |
|:-----|:---|:-----|
| 默认模式 | `light_multi` | 1 primary + 1 low-cost critic |
| 默认启用范围 | Tier2+ | 复杂推理及以上强制启用 |
| 降级策略 | `auto_degrade` | 成本超限时自动降级至 single 模式 |
| 成本预算检查 | 每次调用前估算 | token 估算 + 实时监控 |
| MVP 阶段 | `disabled` via config | 可通过配置完全关闭多模型 |

### 融合策略

- `converger` 模块负责汇总多模型输出
- 支持多数投票、加权融合、排序融合三种策略
- 配置化切换（`config.multillm.converger_strategy`）

### 风险控制

- **响应延迟**：异步调用 + 缓存机制
- **成本估算误差**：预留 20% 余量 + 实时监控告警
- **API 依赖脆弱性**：完善 fallback + 重试策略
- **融合不一致性**：迭代优化提示词 + 解析逻辑

```
        └── mypy
```

## 9. 文档与迭代机制

### 文档维护策略

```
半自动工作流：

cross_validate() 结果
        │
        ▼
脚本自动生成结构化 JSON 摘要  ◄── 每日/每次 crossval 运行
        │
        ▼
脚本生成 Markdown 更新提案
        │
        ▼
开发者手动审查 + 合并到正式文档  ◄── 关键步骤，不可跳过
        │
        ▼
正式文档 ARCHITECTURE_SPEC.md / ENGINEERING_BLUEPRINT.md
```

### 文档层次

| 级别 | 内容 | 更新频率 | 维护方式 |
